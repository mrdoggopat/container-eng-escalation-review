### 1. CONS-8086 — Increasing Memory Usage for CCR When Running SQLServer
- **Link:** [CONS-8086](https://datadoghq.atlassian.net/browse/CONS-8086)
- **Summary:** Cluster check runners (CCRs) running SQLServer checks alongside KSM Core were OOMing. Investigation revealed that KSM Core's large label set from SQL Server pods was causing the high memory usage.
- **Escalation Reason:** Performance / memory regression — intermittent OOM restarts across multiple CCRs.
- **Duration:** 29 days (Feb 12 → Mar 13)
- **Assignee:** Mathieu Colin | **Reporter:** Sean Kornatowski
- **Status Flow:** TEE Triage → TEE - In Progress → Waiting for TSE _(×several)_ → Engineering Triage → Engineering - In Progress → Waiting for TSE → Done → Archive
- **Preventable:** **Yes** — Alexandre Lavigne (engineering) commented with the root cause (KSM Core tag processing from SQLServer labels) on Mar 6, the same day as escalation. This insight could potentially have been obtained via a Slack consultation with the Agent/KSM team without a formal Engineering Triage escalation.
- **Solution Category:** Suggestion without PR or code fixes (engineering analysis; customer confirmed improvement after configuration changes).
- **Non-TEE Engineering Involvement:**
  - Binita Poudel (TSE relay) — updated the card with customer flares and responses throughout.
  - Alexandre Lavigne (engineering) — identified KSM Core label processing as the root cause; provided investigation notes.
- **Improvement Recommendations:** A Slack channel pattern or internal KB entry covering CCR + KSM Core memory interactions could enable TEE to resolve such cases without formal escalation.

### 2. CONS-8117 — Container crash with Datadog APM enabled (Windows 2025, .Net, ECS)
- **Link:** [CONS-8117](https://datadoghq.atlassian.net/browse/CONS-8117)
- **Summary:** Docker Daemon on Windows 2025 was crashing with a Go runtime double-panic when the Datadog agent was enabled. Engineering determined the agent was triggering a code path in an older Docker version that caused a `fatal error: panic during panic` in the garbage collector. Customer resolved by upgrading Docker.
- **Escalation Reason:** Bug / compatibility — production crashes caused by Datadog agent on Windows 2025 + .Net ECS tasks.
- **Duration:** 13 days (Feb 24 → Mar 9)
- **Assignee:** Alexandre Lavigne | **Reporter:** Hassan Albujasim

- **Solution Category:** Customer environment specific issue (old Docker version incompatible with Windows 2025; resolved by upgrading Docker).
- **Non-TEE Engineering Involvement:**
  - Yiyuan Geng (TSE relay) — relayed customer updates while reporter was out.

## Improvement Recommendations

1. **KSM + CCR memory runbook** — CONS-8086 revealed a gap: TEE lacks a quick reference for diagnosing memory growth in CCRs co-running KSM Core checks. A 1-page internal KB note on KSM label cardinality + CCR sizing would prevent re-escalation for this pattern.

2. **GKE Autopilot runbook** — CONS-8154 was resolved by TEE alone. Adding a GKE Autopilot runbook (WorkloadAllowlist policy overrides, known Helm values) would prevent future Engineering Triage escalations for this class.

3. **Operator release notes / migration guides** — CONS-8181 and CONS-8186 both involved Operator behavior changes. Structured release-note callouts for breaking configuration changes (especially in introspection and reconciliation features) would reduce confusion on upgrades.

4. **APM SSI + Istio cross-team path** — CONS-8159 shows APM TSE escalating to Containers, then Containers escalating to Engineering. A shared Slack channel or direct escalation path between APM TSE and Containers Engineering would cut the chain.

5. **Windows agent compatibility matrix** — CONS-8117 involved an agent version × Docker version × Windows 2025 incompatibility. A living compatibility table would let TEE short-circuit diagnosis for Windows ECS crash cases.

6. **vCluster documentation** — CONS-8203 is the first vCluster case in this period. With vCluster adoption growing, a dedicated integration guide (log collection, Kubelet API compatibility, `useApiServer` guidance) should be created proactively.
